工作流引擎 — 常见问题解答 ​

基础概念 ​

Q: 工作流引擎支持哪些 BPMN 元素？ ​

A: 当前支持以下元素：

  • 事件: 开始事件 (start_event)、结束事件 (end_event)
  • 任务: 用户任务 (user_task)、审批任务 (approval_task)
  • 网关: 排他网关 (exclusive_gateway)、并行网关 (parallel_gateway)

预留但暂未实现：服务任务 (service_task)、包含网关 (inclusive_gateway)。

Q: 排他网关和并行网关有什么区别？ ​

A:

  • 排他网关 (Exclusive Gateway): 条件分支，只走一条满足条件的路径。类似编程中的 if-else。
  • 并行网关 (Parallel Gateway): 同时执行所有分支路径，所有分支都完成后才继续。类似编程中的多线程并行。

Q: 流程变量是什么？如何使用？ ​

A: 流程变量是与流程实例关联的键值对数据，在整个流程生命周期内可用。用途：

  • 在条件表达式中使用（如 amount > 1000）
  • 在任务分配中引用（如 ${initiator} 获取发起人 ID）
  • 在审批后自动设置 approved = true/false

设置方式：

ruby
engine.start!(initiator: user, variables: { amount: 5000, department: "IT" })
engine.update_variables(approved: true)

Q: 如何设计多级审批？ ​

A: 使用串行审批模式，在 ApprovalService.submit 中传入多个审批人：

ruby
Workflow::ApprovalService.submit(
  definition_key: "approval_flow",
  initiator: current_user,
  title: "采购申请",
  approvers: [manager, director, cfo]  # 按顺序审批
)

第一个审批人通过后，自动创建下一个审批人的任务。

流程设计 ​

Q: 如何设置任务的截止时间？ ​

A: 在节点的 due_duration 属性中使用 ISO 8601 持续时间格式：

  • P1D — 1 天
  • P3D — 3 天
  • PT2H — 2 小时
  • PT30M — 30 分钟

当前实现仅支持天级别的截止时间计算。

Q: 条件表达式支持哪些语法？ ​

A: 支持简单的二元比较表达式：

variable operator value

操作符: ==, !=, >, <, >=, <=

值类型:

  • 布尔: true, false
  • 数字: 1000, 3.14
  • 字符串: "approved", 'rejected'
  • 变量名: 直接引用流程变量的值

示例:

approved == true
amount > 1000
status != "rejected"

Q: 如何实现条件分支后的不同处理？ ​

A: 使用排他网关，为每条出边设置条件：

json
{
  "nodes": [
    { "id": "gateway", "type": "exclusive_gateway", "outgoing": ["f_approve", "f_reject"] }
  ],
  "flows": [
    { "id": "f_approve", "source": "gateway", "target": "approved_task", "condition": "approved == true" },
    { "id": "f_reject", "source": "gateway", "target": "rejected_task" }
  ]
}

注意：没有条件的流会被视为默认路径。

Q: 如何让任务自动分配给特定用户？ ​

A: 在节点属性中设置 assignee：

  • 固定用户: 直接写用户 ID，如 "assignee": "42"
  • 动态分配: 使用变量引用，如 "assignee": "${initiator}"（分配给发起人）

任务操作 ​

Q: 审批驳回后流程会怎样？ ​

A: 使用 ApprovalService.reject 时：

  1. 当前任务状态变为 cancelled
  2. 流程实例状态变为 failed
  3. 记录驳回历史
  4. 通知发起人被驳回

驳回后流程终止，不会继续推进。

Q: 如何撤回已发起的审批？ ​

A: 只有发起人可以撤回，且只能撤回 running 状态的流程：

ruby
Workflow::ApprovalService.withdraw(instance: instance, user: initiator)

撤回后所有待办任务被取消，流程实例状态变为 cancelled。

Q: 任务转办后原处理人还能操作吗？ ​

A: 不能。转办后任务的 assignee 变为新用户，原处理人失去操作权限。

Q: 催办会做什么？ ​

A: 调用 TaskService.remind 会向任务当前处理人发送一条通知消息，提醒其处理待办任务。 催办本身不会改变任务状态。

技术问题 ​

Q: BPMN 定义的 JSON 格式是什么？ ​

A: 包含 nodes 和 flows 两个数组：

json
{
  "nodes": [
    { "id": "string", "type": "node_type", "name": "string", ... }
  ],
  "flows": [
    { "id": "string", "source": "node_id", "target": "node_id", "condition": "string" }
  ]
}

详细字段说明参见 架构文档 和 设计器指南。

Q: 如何调试流程执行？ ​

A:

  1. 查看 workflow_histories 表，记录了所有执行步骤
  2. 实例详情页 (/admin/workflow/instances/:id) 展示完整的时间线
  3. 检查 workflow_instances.variables 确认流程变量是否正确
  4. 查看 Rails 日志中的异常信息

Q: 流程定义可以版本化吗？ ​

A: 可以。WorkflowDefinition 支持版本管理：

  • key + version 组成唯一约束
  • 调用 create_new_version 创建新版本
  • latest_version(key) 获取最新版本
  • active 作用域只返回已启用的定义

Q: 并行网关的汇聚（join）如何处理？ ​

A: 当前实现中，汇聚逻辑标记为 TODO。并行分支各自独立执行到结束事件。 完整的汇聚实现需要在 execute_end_event 中检查所有并行分支是否完成。

Q: 如何扩展支持新的节点类型？ ​

A:

  1. 在 Parser::NODE_TYPES 中添加新类型
  2. 在 Executor 中添加对应的 execute_xxx_task 方法
  3. 在 Engine#execute_node 的 case 语句中添加分支
  4. 在前端设计器中添加节点组件

Q: 通知服务如何集成？ ​

A: 引擎通过 NotificationService.notify 发送通知。需要确保项目中已实现 NotificationService。 通知类型包括：任务分配、审批待办、审批结果、任务催办等。

Q: 数据库性能优化建议？ ​

A:

  • workflow_histories 表会快速增长，建议定期归档旧数据
  • workflow_instances 已在 status 和 started_at 上建索引
  • workflow_tasks 在 assignee_id 和 status 上可按需添加复合索引
  • 大量实例时考虑按时间分区

故障排除 ​

Q: 保存流程时报错"缺少开始事件" ​

A: 确保 BPMN 定义中至少有一个 type: "start_event" 的节点。

Q: 流程启动后没有生成任务 ​

A: 检查：

  1. 开始事件后面是否正确连接了 user_task 或 approval_task 节点
  2. 流（flow）的 source 和 target 是否正确
  3. 查看是否有解析错误

Q: 条件网关没有匹配到任何分支 ​

A: 检查：

  1. 条件表达式语法是否正确
  2. 流程变量中是否有对应的变量
  3. 变量值是否与条件匹配
  4. 是否设置了默认流（default_flow）

Q: 审批通过后流程没有继续 ​

A: 检查 candidate_users 是否正确配置了所有审批人，以及当前审批人是否在列表中。
